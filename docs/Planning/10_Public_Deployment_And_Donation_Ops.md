# 公网部署与捐助运营架构备忘

**文件路径:** `docs/10_Public_Deployment_And_Donation_Ops.md`  
**状态:** 产品/架构备忘 · 后续开发指导  
**最后更新:** 2026-04-21

---

## 1. 本文目的

本文记录当前工程能否直接公网部署运营的评估结论，并为后续产品开发提供方向。

当前目标不是立刻做完整商业化发行，而是先形成一个最小可接受的私域收入闭环：

1. 玩家可以访问公网地址游玩。
2. 玩家通过微信群、QQ群与作者建立联系。
3. 玩家自愿捐助。
4. 作者通过后台管理工具，为捐助者发放通宝、沙漏、权限或其他权益。
5. 玩家权益能够稳定保存，且不依赖玩家本地浏览器存档。

---

## 2. 当前结论

当前工程可以部署为公网试玩 Demo，但还不能作为可收费运营版本。

原因很直接：

- 当前项目是纯前端 React/Vite 单页应用。
- 游戏存档保存在浏览器 `localStorage` 中。
- 没有账号系统。
- 没有服务端数据库。
- 没有后台管理站。
- 没有服务端校验通宝、沙漏、冷却、任务结算等关键经济行为。
- 玩家可以在浏览器开发者工具中修改本地存档，从而改等级、铜钱、通宝、沙漏、装备和任务状态。

所以，当前版本适合：

- 公网试玩。
- 收集玩家反馈。
- 验证题材、美术、UI、核心循环。
- 验证玩家是否愿意加入私域群。
- 验证通宝、沙漏等高级资源是否能形成捐助动机。

当前版本不适合：

- 正式承诺付费权益。
- 长期保存玩家资产。
- 由后台给玩家发放通宝。
- 防止玩家篡改资源。
- 跨设备同步存档。
- 处理大量玩家运营。

---

## 3. 当前工程的有利条件

虽然还不能直接收费运营，但工程并不是走错方向。它已经具备一些有利于后续扩展的基础。

### 3.1 静态部署成本低

当前项目已有 Vite 构建流程，可以通过 Cloudflare Pages、Vercel、Netlify 或普通 Nginx 静态站点部署。

这对早期验证非常合适：

- 上线快。
- 维护成本低。
- 没有服务器运维压力。
- 出问题也不会影响真实付费资产。

### 3.2 GameState 已经集中定义

`GameState`、玩家资源、装备、任务、副本、竞技场等主要数据结构集中在 `src/core/gameState.ts`。

这意味着将来迁移到后端时，可以把现有 `GameState` 作为数据库 schema 和 API DTO 的原型，而不是从零设计。

### 3.3 核心计算已有拆分

项目已经把部分核心逻辑拆到了：

- `src/core/mathCore.ts`
- `src/core/battleCore.ts`
- `src/core/equipmentGenerator.ts`
- `src/data/`

这有利于后续做“前后端共享逻辑”或“迁移到服务端权威结算”。

### 3.4 高级经济资源已经进入设计

通宝、沙漏、任务加速、竞技场跳过冷却、黑市刷新等付费节奏已经在设计和部分实现中出现。

这说明项目已经有明确的私域捐助运营方向，而不是单纯做一个无商业闭环的单机 Demo。

---

## 4. 当前工程的主要风险

### 4.1 本地存档不能承载付费权益

通宝、沙漏、等级、装备都在浏览器本地。

这带来几个问题：

- 玩家可以手动改。
- 玩家清缓存会丢档。
- 换电脑、换浏览器无法继承。
- 作者无法远程修改某个玩家的存档。
- 作者无法确认某个存档属于哪个捐助者。

只要付费权益还在 `localStorage`，就不能称为可运营资产。

### 4.2 前端不能作为经济系统的裁判

当前很多关键操作由前端直接修改 `GameState`：

- 完成任务。
- 获得奖励。
- 消耗通宝。
- 消耗沙漏。
- 刷新黑市。
- 竞技场结算。
- 冷却跳过。
- 背包变化。

这在单机阶段没问题，但进入运营阶段后，前端只能负责展示和发起请求，不能负责最终裁决。

真正的资源变化应该由服务端完成。

### 4.3 开发测试入口不能进入公网版本

当前工程里存在开发期作弊按钮和清档按钮。

这类入口对于开发很方便，但公网部署前必须满足至少一个条件：

- 只在开发环境显示。
- 只对管理员显示。
- 或者在公开试玩包中彻底关闭。

否则它会破坏玩家对资源和捐助权益的信任。

### 4.4 没有后台操作审计

私域捐助运营里，“作者手动给玩家加通宝”是核心动作。

这个动作必须可追踪：

- 谁操作。
- 操作了哪个玩家。
- 加了多少通宝。
- 为什么加。
- 操作前是多少。
- 操作后是多少。
- 对应哪笔捐助或哪条群内记录。

没有审计，早期玩家少时还能靠记忆，玩家稍微多一点就会混乱。

## 5. 建议的阶段路线

### 阶段一：公网试玩版

目标是尽快让真实玩家玩到，并验证兴趣。

这个阶段可以继续使用纯前端静态部署，但需要明确它是试玩版本。

建议目标：

- 部署到公网。
- 页面内放微信群、QQ群加入方式。
- 移除或隐藏开发作弊按钮。
- 降低新玩家默认通宝、沙漏数量，避免误判付费节奏。
- 在群内说明当前为测试服，存档可能调整或清理。
- 不承诺长期资产安全。

这个阶段不急着做完整后端，重点是验证：

- 玩家是否愿意玩超过 10 分钟。
- 玩家是否愿意第二天回来。
- 玩家是否愿意加入群。
- 玩家是否理解通宝、沙漏价值。
- 玩家是否愿意为支持作者而捐助。

### 阶段二：最小收入闭环

目标是支持“捐助者权益可发放、可保存、可查询”。

这是从试玩 Demo 到可运营产品的关键阶段。

必须新增：

- 账号系统。
- 云端玩家存档。
- 后台管理页面。
- 管理员登录。
- 玩家查询。
- 通宝/沙漏发放。
- 权限标记。
- 操作日志。

推荐最小功能：

1. 玩家登录。
2. 玩家 `GameState` 保存到云端。
3. 管理后台可以按玩家 ID、昵称、邀请码、群备注搜索玩家。
4. 管理后台可以加减通宝、沙漏。
5. 管理后台可以设置权限标记，例如 `supporter`, `tester`, `banned`。
6. 所有后台操作写入日志。

这个阶段仍然可以允许部分玩法在前端计算，但所有高级资源变化必须由服务端保存。

### 阶段三：服务端权威经济系统

目标是让游戏能更长期运营，降低作弊和资源污染。

逐步迁移到服务端裁决：

- 任务开始。
- 任务完成。
- 通宝跳过任务。
- 沙漏减少任务时间。
- 黑市刷新。
- 商品购买。
- 竞技场冷却。
- 竞技场结算。
- 每日限制刷新。
- 掉落判定。

前端在这个阶段只负责：

- 展示状态。
- 发起操作请求。
- 播放动画。
- 展示服务端返回结果。

服务端负责：

- 校验玩家身份。
- 校验资源是否足够。
- 扣除资源。
- 发放奖励。
- 写入数据库。
- 返回结算结果。

## 6. Supabase 与 MongoDB Atlas 的选择建议

作者已有 Supabase 和 MongoDB Atlas 账号。两者都可以用，但适合的路线不同。

### 6.1 推荐优先方案：Supabase

如果目标是尽快做出最小收入闭环，优先推荐 Supabase。

原因：

- 自带 Auth，适合快速做账号系统。
- 自带 Postgres，适合保存玩家、资源、捐助记录、后台操作日志。
- 自带 Row Level Security，方便做玩家只能访问自己存档。
- 后台管理可以直接基于数据库表和 API 开发。
- 适合小团队、单人作者、早期私域运营。

Supabase 适合的表：

- `profiles`: 玩家基础信息。
- `player_saves`: 玩家云端存档。
- `player_resources`: 通宝、沙漏、铜钱、声望等关键资源。
- `donation_records`: 捐助记录。
- `admin_actions`: 后台操作日志。
- `access_invites`: 邀请码或访问 Token。
- `player_flags`: 玩家权限或状态标记。

### 6.2 MongoDB Atlas 的适合场景

MongoDB Atlas 也可以用，尤其适合直接保存完整 `GameState` 文档。

优点：

- `GameState` 是嵌套结构，MongoDB 存起来很自然。
- 存档 schema 频繁变化时，文档数据库更灵活。
- 装备、背包、任务状态这类复杂对象可以直接保存。

风险：

- 账号系统、权限控制、后台审计通常需要自己搭更多东西。
- 对最小收入闭环来说，上手成本可能比 Supabase 高。
- 如果没有额外后端服务，前端直连 MongoDB 不合适。

### 6.3 实用建议

早期建议：

- Supabase 负责账号、权限、资源、后台、审计。
- 完整 `GameState` 可以先作为 JSONB 存在 Supabase Postgres 中。
- 如果后期存档结构越来越复杂，再考虑 MongoDB Atlas 专门承载详细存档。

也就是说，不必一开始就在 Supabase 和 MongoDB 之间做最终选择。

最稳妥的第一步是用 Supabase 做最小闭环。

## 7. 最小后端数据模型草案

以下是概念模型，不是最终数据库设计。

### 7.1 玩家表

`profiles`

- `id`: 用户 ID。
- `display_name`: 游戏昵称。
- `qq_name`: QQ 群备注，可选。
- `wechat_name`: 微信群备注，可选。
- `created_at`: 注册时间。
- `last_login_at`: 最近登录时间。
- `status`: `active`, `banned`, `deleted`。

### 7.2 玩家资源表

`player_resources`

- `player_id`: 玩家 ID。
- `copper`: 铜钱。
- `prestige`: 声望。
- `rations`: 干粮或精力。
- `tokens`: 通宝。
- `hourglasses`: 沙漏。
- `updated_at`: 最近更新时间。

关键点：

- 通宝、沙漏必须服务端保存。
- 前端展示的数值必须来自服务端。
- 后台修改资源必须写日志。

### 7.3 玩家存档表

`player_saves`

- `player_id`: 玩家 ID。
- `save_version`: 存档版本。
- `game_state`: 完整 GameState JSON。
- `updated_at`: 最近保存时间。

早期可以用完整 JSON 保存，减少重构成本。

后期再把高价值字段拆出来独立建表，例如：

- 装备。
- 背包。
- 任务。
- 竞技场。
- 副本进度。
- 每日限制。

### 7.4 捐助记录表

`donation_records`

- `id`: 捐助记录 ID。
- `player_id`: 玩家 ID。
- `amount`: 捐助金额，可选。
- `channel`: 微信、QQ、支付宝、其他。
- `note`: 备注。
- `reward_tokens`: 发放通宝数量。
- `operator_id`: 操作管理员。
- `created_at`: 创建时间。

### 7.5 后台操作日志表

`admin_actions`

- `id`: 日志 ID。
- `operator_id`: 管理员 ID。
- `target_player_id`: 被操作玩家 ID。
- `action_type`: 操作类型。
- `before`: 操作前快照。
- `after`: 操作后快照。
- `reason`: 操作原因。
- `created_at`: 操作时间。

这张表非常重要。它不是锦上添花，而是私域运营的安全绳。

## 8. 前端迁移策略

不要一次性把所有玩法都推翻重写。

推荐用渐进迁移方式。

### 8.1 引入存档适配层

当前 UI 直接调用 `loadGameState()` 和 `saveGameState()`。

后续可以设计一层 `SaveAdapter` 或 `GameApi`，让 UI 不关心数据来自本地还是云端。

概念上分两种实现：

- `LocalSaveAdapter`: 继续读写 `localStorage`，用于开发和离线试玩。
- `RemoteSaveAdapter`: 调用 Supabase 或后端 API，保存到云端。

这样可以减少一次性重构风险。

### 8.2 先迁移高价值资源

不必第一天就把整个游戏都服务端化。

优先迁移：

- `tokens`
- `hourglasses`
- 捐助权限
- 管理后台发放记录

其次迁移：

- 任务完成。
- 商店刷新。
- 竞技场冷却。
- 每日限制。

最后迁移：

- 战斗回放动画。
- UI 状态。
- 临时提示。

### 8.3 前端仍可保留预测与动画

服务端权威并不意味着前端什么都不能算。

前端可以继续做：

- 倒计时显示。
- 战斗动画播放。
- 奖励预览。
- 装备面板计算展示。
- 本地缓存。

但最终结果必须以服务端返回为准。

## 9. 后台管理站最小需求

第一版后台不要做复杂，能解决运营动作即可。

### 9.1 必须功能

- 管理员登录。
- 玩家列表。
- 玩家搜索。
- 查看玩家基础信息。
- 查看玩家当前资源。
- 给玩家增加或扣除通宝。
- 给玩家增加或扣除沙漏。
- 修改玩家权限标记。
- 查看最近后台操作日志。

### 9.2 建议功能

- 给玩家备注 QQ 群名、微信群名。
- 记录捐助渠道。
- 记录捐助金额。
- 一键复制玩家 ID。
- 封禁玩家。
- 恢复玩家云存档。

### 9.3 不建议第一版做的功能

- 复杂 GM 命令系统。
- 全量装备编辑器。
- 大型客服系统。
- 自动支付回调。
- 多管理员复杂权限矩阵。

早期的核心是能安全发放权益，不是把后台做成完整运营平台。

## 10. 上线前检查清单

### 10.1 公网试玩版检查

- 已构建静态产物。
- 页面能正常访问。
- 移除或隐藏开发作弊按钮。
- 移除或隐藏清档按钮，或只保留在明显的测试入口。
- 明确告诉玩家当前是测试版。
- 页面中有微信群、QQ群入口。
- 初始通宝、沙漏数量符合测试目的。
- 已准备反馈收集渠道。

### 10.2 最小捐助运营版检查

- 玩家有账号。
- 玩家有唯一 ID。
- 玩家资源保存在云端。
- 通宝、沙漏不能只存在 `localStorage`。
- 管理员可以登录后台。
- 管理员可以搜索玩家。
- 管理员可以发放通宝。
- 管理员操作会写入日志。
- 玩家刷新页面后能看到云端资源。
- 换设备登录后能恢复权益。

### 10.3 长期运营版检查

- 关键经济行为由服务端校验。
- 每日限制由服务端记录。
- 冷却时间由服务端记录。
- 掉落和奖励由服务端裁决。
- 有异常资源变动审计。
- 有基本备份策略。
- 有封禁和恢复手段。

## 11. 产品开发原则

### 11.1 先验证玩家愿不愿意留下

技术架构很重要，但早期最重要的是验证：

- 核心循环是否有吸引力。
- 题材是否成立。
- 玩家是否愿意每天回来点几次。
- 玩家是否愿意加入群。
- 玩家是否愿意支持作者。

如果这些不成立，完整后端也救不了产品。

### 11.2 付费权益必须可信

只要涉及通宝、沙漏、捐助奖励、权限功能，就必须逐步脱离纯前端存档。

哪怕是私域小圈子，也要保护捐助者的信任。

### 11.3 后台是收入闭环的一部分

后台不是附属功能，而是当前商业模式的核心功能。

因为当前模式不是自动充值，而是：

玩家捐助 -> 作者确认 -> 作者手动发放权益 -> 玩家继续游玩。

如果没有后台，这个闭环就只能靠人工改本地文件或口头承诺，无法长期稳定。

### 11.4 不要过早追求大型 MMO 架构

本项目当前更接近“私域运营的放置 RPG”，不是万人同屏网游。

早期架构目标应该是：

- 够用。
- 可迁移。
- 能审计。
- 能发放权益。
- 不阻碍玩法迭代。

不要一开始就为了反外挂、分布式、复杂战斗服务器而拖慢产品验证。

## 12. 建议的近期优先级

从产品价值看，建议后续优先级如下：

1. 完成公网试玩部署。
2. 隐藏开发作弊入口。
3. 增加群入口与测试说明。
4. 观察玩家试玩反馈。
5. 设计 Supabase 最小账号与云存档方案。
6. 做最小后台：查玩家、发通宝、写日志。
7. 把通宝、沙漏迁移到云端权威存储。
8. 再继续扩展马厩、转盘、宠物、更多副本等内容。

一句话原则：

先让玩家愿意留下，再让捐助权益可信，最后再逐步加强服务端权威。
