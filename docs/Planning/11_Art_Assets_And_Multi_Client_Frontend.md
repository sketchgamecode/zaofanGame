# 美术资源与多端前端路线备忘

**文件路径:** `docs/Planning/11_Art_Assets_And_Multi_Client_Frontend.md`  
**状态:** 产品/架构备忘 · 后续开发指导  
**最后更新:** 2026-04-21

---

## 1. 本文目的

当前游戏前端仍然比较简陋，更像一个功能型 Web App，而不是 Shakes & Fidget 那种以场景、角色、装备、道具、背景和按钮热点构成的图像化游戏界面。

本文记录以下问题的阶段性判断：

1. 美术资源应该在哪个阶段引入。
2. 前后端分离时，是否需要提前考虑美术资源植入。
3. 一个游戏服务器是否可以支撑多个不同前端。
4. PC 浏览器、手机浏览器、手机 App、Steam PC 版是否应该共用同一个前端。
5. 如何避免早期过度工程化，同时不堵死未来多端运营路线。

---

## 2. 核心判断

长期看，本项目不应该只有一个前端。

更合理的目标是：

- 一个统一的游戏服务端。
- 一套统一的账号、存档、经济、权限、运营后台。
- 多个针对不同平台体验优化的客户端。

这些客户端可以包括：

- PC 浏览器版。
- 手机浏览器版。
- 手机 App 版。
- Steam PC 版。

但是，短期内不应该立刻为所有平台分别开发完整客户端。

早期最优路线是：

1. 先做一个响应式 Web 前端，完成玩法验证和私域运营验证。
2. 在架构上提前区分“游戏数据/API”和“表现层资源/UI”。
3. 当核心循环、收入闭环、美术方向稳定后，再拆出更专业的移动端或 Steam 端。

---

## 3. 当前前端的问题

当前前端更接近“管理面板式 UI”，优点是开发快、结构清楚、便于验证玩法。

但它和目标体验存在明显差距：

- 缺少完整场景背景。
- 缺少角色立绘或头像体系。
- 缺少装备图标和道具图标。
- 缺少 S&F 式热点场景交互。
- 缺少强风格化按钮、面板、弹窗。
- 移动端和 PC 端交互还没有明确分化。
- 视觉资产尚未形成可管理的资源库。

这不代表当前前端没有价值。它的价值是快速验证系统闭环。

但一旦进入对外试玩和捐助运营，前端观感会显著影响玩家信任感和付费意愿。

---

## 4. 美术资源应该分阶段引入

### 4.1 阶段一：玩法验证期

目标是验证游戏是否好玩，不追求完整美术。

建议使用：

- 临时背景图。
- 少量关键图标。
- 简单装备占位图。
- 少量角色头像。
- 统一色调和基础面板风格。

这个阶段不建议投入大量美术成本。

原因：

- 玩法还会调整。
- 模块还会增删。
- UI 信息架构还不稳定。
- 过早制作大量资源容易返工。

但有几类资源可以提前准备：

- Logo。
- 主城或客栈主背景。
- 通宝图标。
- 沙漏图标。
- 铜钱图标。
- 默认角色头像。
- 默认装备图标模板。

这些资源对玩家第一印象影响大，且后续复用概率高。

### 4.2 阶段二：公网试玩期

目标是让玩家觉得“这是一个游戏”，而不是“一个开发中的表单 App”。

建议优先补齐：

- 主城/客栈背景。
- 黑市场景背景。
- 竞技场背景。
- 副本战斗背景。
- 角色职业头像。
- 装备品质图标框。
- 通宝、沙漏、铜钱、声望图标。
- 任务 NPC 或江湖客头像。
- 主要按钮和弹窗皮肤。

这个阶段美术资源不需要特别多，但要让核心循环的每一步都有视觉反馈。

重点不是数量，而是“核心路径完整”：

进入游戏 -> 查看角色 -> 进入客栈 -> 接任务 -> 等待/跳过 -> 战斗 -> 领奖 -> 消耗通宝。

### 4.3 阶段三：最小收入闭环期

目标是支撑捐助转化。

建议增加：

- 捐助者权益展示图标。
- 稀有装备图标。
- 高级资源购买/发放反馈。
- 更精致的任务结算图。
- 更清楚的通宝消耗动画。
- 群入口或作者公告区域的视觉包装。

此阶段的美术要服务于信任和价值感。

玩家愿意捐助，不只是因为数值有用，也因为他感觉这个项目有生命力、作者在认真运营、自己的支持能换来明确体验。

### 4.4 阶段四：多端扩展期

目标是为不同平台建立适配体验。

此时再考虑：

- 移动端专用布局。
- Steam 版宽屏布局。
- App 启动画面。
- 平台专属图标和商店截图。
- 更完整的角色纸娃娃或装备外观系统。
- 大量副本背景和怪物立绘。

这个阶段才适合系统性投入大规模美术资源。

## 5. 前后端分离时如何考虑美术资源

美术资源不应该进入游戏服务端核心逻辑。

游戏服务端应该关心：

- 玩家是谁。
- 玩家有什么资源。
- 玩家处于什么状态。
- 玩家能不能执行某个操作。
- 操作后资源如何变化。
- 哪个装备、道具、怪物、任务被引用。

前端或资源服务应该关心：

- 这个装备显示什么图。
- 这个怪物使用哪张立绘。
- 这个场景加载哪张背景。
- 这个按钮是什么风格。
- 这个平台应该用哪套布局。

也就是说，服务端返回的应该是稳定 ID，而不是直接绑定某个前端图片路径。

示例概念：

- 服务端返回 `item_id: "weapon_green_saber_001"`。
- 前端根据资源清单把它映射为某张图标、名称、品质框和描述。
- PC 前端可以显示大图标。
- 手机前端可以显示小图标。
- Steam 前端可以显示高清资源。

这样做的好处：

- 同一套服务端可以服务多个前端。
- 美术资源可以独立更新。
- 不同平台可以使用不同分辨率资源。
- 不会因为换前端而改服务端经济逻辑。

## 6. 一个游戏服务器支撑多个前端是合理方向

长期看，应该以“单一权威游戏服务端 + 多客户端”为目标。

统一服务端负责：

- 登录。
- 存档。
- 通宝、沙漏和经济系统。
- 任务、竞技场、副本结算。
- 后台发放权益。
- 权限与封禁。
- 活动配置。

不同前端负责：

- 布局。
- 操作方式。
- 图片资源。
- 动画表现。
- 平台能力接入。

这意味着：

- PC 浏览器可以做大场景、横向布局、鼠标悬浮信息。
- 手机浏览器可以做纵向布局、大按钮、少悬浮、多弹层。
- 手机 App 可以利用推送、本地缓存、全屏体验。
- Steam PC 版可以做更沉浸的桌面窗口、成就、商店页和启动器体验。

核心原则：

不要让服务端知道“这个玩家用的是哪个 UI 组件”。

服务端只知道“这个玩家请求执行某个游戏动作”，然后返回结果。

---

## 7. 是否需要多个完全不同前端

短期不需要。

早期建议用一个响应式 Web 前端覆盖 PC 浏览器和手机浏览器。

原因：

- 开发成本最低。
- 玩法还没完全稳定。
- 美术规格还没完全确定。
- 私域验证主要看留存和捐助意愿，不需要一开始覆盖所有平台。

但中长期应该承认：

一个前端很难同时做到 PC、手机、App、Steam 都体验优秀。

尤其是本项目如果走 S&F 风格，会有大量：

- 场景热点。
- 背景图裁切。
- 横向构图。
- 弹窗。
- 装备格子。
- 战斗表现。
- 图标密集信息。

这些内容在 PC 横屏和手机竖屏上天然冲突。

所以更合理的中长期路线是：

1. 第一阶段：一个响应式 Web 前端。
2. 第二阶段：在同一代码库中拆出 PC 布局和 Mobile 布局。
3. 第三阶段：如果移动端玩家明显更多，再做 App 壳或独立移动端。
4. 第四阶段：如果 PC 付费玩家稳定，再考虑 Steam 版。

## 8. 建议的前端分层

为了未来多端不痛苦，前端可以逐步分成几层。

### 8.1 Game API 层

负责和服务端通信。

包括：

- 登录。
- 拉取存档。
- 保存存档。
- 开始任务。
- 完成任务。
- 消耗通宝。
- 刷新黑市。
- 领取奖励。

这一层不应该包含 UI。

### 8.2 Game Domain 层

负责前端可复用的游戏领域模型。

包括：

- `GameState` 类型。
- 资源类型。
- 装备类型。
- 任务类型。
- 战斗回放事件类型。

这一层可以被 PC Web、Mobile Web、App 复用。

### 8.3 Asset Registry 层

负责把游戏 ID 映射到资源。

包括：

- 装备图标。
- 道具图标。
- 背景图。
- NPC 头像。
- 怪物立绘。
- UI 音效。

这一层可以根据平台选择不同资源规格。

例如：

- PC 使用高清横图。
- Mobile 使用裁切后的竖屏图。
- App 使用本地缓存资源。

### 8.4 UI Layout 层

负责具体平台的界面。

例如：

- PC 客栈界面。
- Mobile 客栈界面。
- PC 战斗界面。
- Mobile 战斗界面。

同一个游戏状态，可以有不同 UI 解释。

### 8.5 Presentation Effects 层

负责动画、音效、粒子、转场。

这一层最容易因平台而不同，不应和经济逻辑绑定。

## 9. 美术资源管理建议

### 9.1 建立资源命名规则

从早期就应该避免随便命名图片。

建议命名结构：

```text
bg_tavern_main_v001.png
bg_arena_day_v001.png
icon_currency_token_v001.png
icon_currency_hourglass_v001.png
item_weapon_saber_green_001.png
item_armor_chest_blue_001.png
npc_questgiver_male_001.png
monster_bandit_001.png
ui_panel_wood_dark_v001.png
```

命名要表达：

- 类型。
- 用途。
- 主题。
- 品质或变体。
- 版本。

### 9.2 建立资源清单

建议未来有一个资源清单文件，记录：

- 资源 ID。
- 文件路径。
- 适用平台。
- 分辨率。
- 用途。
- 版权来源。
- 是否可商用。
- 是否临时占位。

这对 AI 生成图、美术外包、后期替换都很重要。

### 9.3 区分占位资源和正式资源

早期可以大量使用占位图，但必须标记清楚。

建议分类：

- `placeholder`: 临时占位。
- `prototype`: 原型期可用。
- `production`: 正式资源。
- `licensed`: 授权资源。
- `ai_generated_review_needed`: AI 生成但待审核。

### 9.4 不要让图片路径散落在组件中

前端组件不应该到处写死图片路径。

更好的方式是：

- 组件拿到 `assetId`。
- 通过 Asset Registry 查询图片。
- 渲染对应资源。

这样未来换图、换平台、换分辨率不会到处改组件。

## 10. 对服务端 API 的影响

为了支持多前端，服务端 API 设计需要注意：

### 10.1 返回语义数据，不返回 UI 数据

推荐：

```json
{
  "itemId": "weapon_green_saber_001",
  "quality": "green",
  "slot": "mainHand",
  "level": 12
}
```

不推荐：

```json
{
  "imageUrl": "/pc/assets/items/saber_green_1024.png",
  "buttonColor": "green",
  "x": 120,
  "y": 80
}
```

服务端不应该绑定某个平台的 UI 实现。

### 10.2 活动和配置可以返回资源 ID

如果未来做节日活动或特殊副本，服务端可以返回：

- `backgroundAssetId`
- `npcAssetId`
- `monsterAssetId`
- `rewardIconAssetId`

但具体加载哪张图片，仍由前端资源系统决定。

### 10.3 存档中不要保存具体图片路径

玩家存档应该保存：

- 装备 ID。
- 外观 ID。
- 坐骑 ID。
- 宠物 ID。

不要保存：

- 图片路径。
- CSS class。
- 前端布局参数。

## 11. Steam 与 App 的远期判断

### 11.1 手机 App

手机 App 的价值：

- 更强留存。
- 可以推送任务完成提醒。
- 可以做更沉浸的全屏体验。
- 更适合每日多次打开。

但 App 也带来成本：

- 打包发布。
- 应用商店审核。
- 支付合规。
- 多平台适配。
- 热更新限制。

早期不建议直接做原生 App。

可选路线：

1. 先做手机 Web。
2. 再做 PWA。
3. 再考虑 Capacitor/Tauri/React Native 等壳。

### 11.2 Steam PC 版

Steam 版的价值：

- 增强可信度。
- 获得 PC 玩家入口。
- 可以做成窗口化放置游戏。
- 有机会接 Steam 成就、愿望单和社区。

但 Steam 版需要：

- 更完整的美术表现。
- 更稳定的服务端。
- 更清楚的商业模式。
- 更少的版权和素材风险。
- 更高完成度的新手体验。

Steam 应该是中后期目标，不是早期目标。

## 12. 推荐路线

### 12.1 近期

- 保持一个 Web 前端。
- 继续完成核心玩法。
- 引入少量关键美术资源。
- 建立资源命名规则。
- 建立资源清单。
- 不急着做 App 或 Steam。

### 12.2 中期

- 做公网试玩。
- 接 Supabase 最小账号和云存档。
- 做后台发放通宝。
- 建立 PC/Mobile 响应式布局边界。
- 把图片路径集中到 Asset Registry。
- 核心场景补齐正式或半正式美术。

### 12.3 后期

- 根据真实玩家数据决定是否拆移动端。
- 如果移动端活跃高，优先做 PWA 或 App 壳。
- 如果 PC 玩家捐助强，考虑 Steam 版。
- 服务端保持统一，不为某个前端写死逻辑。

## 13. 开发原则

### 13.1 先让游戏成立，再让美术变重

美术很重要，但不应该早于核心循环稳定。

合理顺序是：

1. 核心循环可玩。
2. 玩家愿意回来。
3. 捐助闭环可信。
4. 美术投入逐步加重。
5. 多端体验分化。

### 13.2 美术资源是产品资产，不是组件装饰

未来会有大量角色、道具、装备、背景和 UI 皮肤。

它们应该被当作资产库管理，而不是散落在组件里的图片引用。

### 13.3 多端共享服务端，不共享所有 UI

多个前端可以共享：

- 账号。
- 存档。
- 经济。
- 配置。
- 战斗结算。
- 权限。

但不一定共享：

- 布局。
- 背景裁切。
- 操作方式。
- 动画节奏。
- 图像分辨率。

### 13.4 服务端只关心规则，前端负责表现

这条原则非常重要。

如果服务端和某一个前端表现绑死，未来做移动端或 Steam 版会很痛苦。

如果服务端只返回语义结果，不返回 UI 细节，多端扩展会顺很多。

---

## 14. 一句话结论

本项目长期应该是“一套游戏服务器，多套平台前端”。

但当前阶段不要急着做多端。先用一个 Web 前端验证核心玩法和捐助闭环，同时从现在开始建立资源 ID、资源清单、API 语义边界和前端分层意识。这样既不会拖慢早期开发，也不会堵死未来接入大量美术资源、移动端和 Steam 版的路线。
