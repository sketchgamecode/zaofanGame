## 一、产品核心定位

### 1.1 机会窗口
「Shakes & Fidget」是一款经过市场验证的浏览器放置类 RPG。该游戏设计精密，商业模式成熟，但其开发商 Playa Games 已从中国市场撤出，且无意回归。这在中文市场留下了一个**有需求、无竞品**的真空地带。

### 1.2 核心战略：精确复刻，换皮分发
- **游戏机制与数值**：一丝不苟地还原 Shakes & Fidget 的全部系统。不做改动，不做「微创新」。S&F 的成功已被市场验证，核心设计本身即是最大的护城河。
- **客户端与包装**：高度不确定，作为实验变量存在。同一套游戏服务端，对接不同的前端（Web、Unity、UE、iOS、Android），使用不同的文案、美术风格、IP 包装，通过市场数据筛选最优方案。
- **目标**：先验证核心玩法留存，再确定最终的商业包装。

## 二、系统架构：权威服务端 + 多端展示层

### 2.1 核心原则

> **服务端是游戏的唯一真相。客户端是服务端的渲染器。**

客户端**不做任何游戏逻辑计算**，它只做两件事：
1. **发送玩家指令**（按下按钮 → HTTP 请求）
2. **渲染服务端返回的结果**（新的 GameState + 战斗日志 → 更新 UI）

这样设计的优势：
- 天然防作弊（客户端无法修改数值）
- 换前端零成本（Unity 客户端和 Web 客户端对接同一套 API）
- 数值调整只在服务端改，无需更新客户端

### 2.3 API 交互模式（通用于所有客户端）

所有游戏动作都通过**一个统一的 Action Endpoint**进行：

**请求：**
```json
POST /api/action
Authorization: Bearer <supabase_jwt>
Content-Type: application/json

{
  "action": "START_MISSION",
  "payload": { "missionType": "short" }
}
```

**响应：**
```json
{
  "success": true,
  "gameState": { /* 完整的最新 GameState */ },
  "log": [
    { "type": "info",   "text": "出发执行短期任务：押送粮草" },
    { "type": "reward", "text": "预计奖励：120 铜钱，380 XP" }
  ]
}
```

客户端拿到 `gameState` 刷新全局状态，用 `log` 显示动画/文字反馈。Unity、Web、CLI 客户端处理的是**完全相同的 JSON**，只是展现方式不同。

## 五、开发纪律（绝不妥协的原则）

1. **服务端永远是真相**：任何看起来像"临时方案"的客户端数值计算，立刻停止，改为请求服务端接口。
2. **API 先于实现**：每次新增功能，先在 `docs/01_API_Reference.md` 里写好接口文档，再写代码。这样任何客户端开发者（包括 Unity 程序员）都能独立并行开发。
3. **S&F 机制不自由发挥**：数值有疑问时，以 [官方 Wiki](https://sf.rheydt.de/en/wiki/index.php?title=Main_Page) 为准，不猜测，不创新。
4. **只有一套环境**：阿里云新加坡节点 `47.79.240.134` 就是生产环境。本地 `npm run dev` 只跑客户端热重载，所有服务端请求全走云端。
5. **不搞局域网测试**：任何局域网或 localhost 的服务端测试方案都是临时方案，产品开发不允许临时方案存在。
