## 0. 项目定位

> **"完全复刻 Shakes & Fidget 经典版 (2011) 的底层机制，以中国古代历史文学大杂烩宇宙为皮，做一款硬核放置 RPG。"**

我们参照并复刻的是 **S&F 经典版（非现代服务器加速版）**。  
所有数值设计应以"对标经典版玩家的消耗节奏"为基准进行校验。

### 0.1 发行模式
- **捐助制私域发行**，由作者亲自向捐助玩家发放访问 Token
- 暂不公开发行，服务于少量高代入感玩家
- 不涉及商业发行，**无 IP 侵权顾虑**，可自由使用任何中国古典文学作品中的人物、场景、器物

### 文案风格边界规则

#### ✅ 允许的写法

1. **古代语境无冲突的现代梗**：梗的内核来自现实，但包装进古代场景后不显违和
   - `屠龙刀：刀刀烈火，兄弟砍我` — 游戏梗，放在兵器铭文里完全自洽
   - `银样蜡枪头：不知为何，使此武器者一般惧内` — 暗讽能力不足，以婉转古典方式呈现
   - `瑶池木瓜：一对奇珍，节度使探爪` — 野史暗梗（安禄山/杨玉环），有典可查

2. **引经据典的反向解构**：把名句/名场面用于反差语境，用荒诞结局制造讽刺感

3. **借古讽今的人性黑色幽默**：克扣工钱的监工、卖友求荣的官员、收保护费的江湖帮派——以古代角色之口说出，不点破现代对应

4. **带宿命感的悲剧美学**：人物结局可以荒诞，但须带有宿命色彩，符合"天道循环"的古典叙事美学

#### ❌ 禁止的写法

1. **现代流行语与网络热词**：打工人、牛马、躺平、内卷、YYDS、绝绝子、破防、emo 等
2. **现代场景与科技概念**：手机、电脑、快递、外卖、996、直播、算法、流量等
3. **破坏沉浸感的元叙事**：不能出现"这是个游戏"之类的第四堵墙打破
4. **跨文明的硬接**：不能让古代中国角色直接说出西方文化梗（允许结构相似的隐晦类比）
5. **当代政治隐喻**：任何可能被解读为当代政治讽刺的内容

#### 文案格式参考模板

**装备描述格式：**
```
【简短对仗定语】正文一到两句，用古白话节奏，末句可带转折或反讽。
```

**Boss 描述格式：**
```
第三人称旁白，交代此人荒诞经历（可引用典故），末句以讽刺或宿命感收尾。
```

## 1. 架构原则

### 1.1 当前架构：纯前端单机
- 游戏目前**没有服务器**，所有逻辑在浏览器（React + TypeScript）中执行。
- 持久化：`localStorage`（Key: `zaofan_game_state_v1`）。
- 计划：**架构代码必须为前后端分离迁移做好准备**。具体做法是将所有核心计算封装为纯函数（`mathCore.ts`, `battleCore.ts`），UI 层不写任何计算逻辑。

### 1.2 数据驱动原则 ⚠️ 严格遵守
- **所有静态配置（Boss 属性、XP 阈值、奖励）必须从 `src/data/` 目录的配置文件中读取。**
- **严禁在 UI 组件或业务逻辑中硬编码任何游戏数值。**
- 配置文件使用 **TypeScript（`.ts`）而非 JSON**，以获得编译期类型检查。

### 1.3 核心文件职责
| 文件 | 职责 | 禁止事项 |
|---|---|---|
| `src/core/mathCore.ts` | 所有数学公式（HP/伤害/暴击/护甲/升级） | 禁止在此写 UI 逻辑 |
| `src/core/battleCore.ts` | 战斗回合模拟，输出 `TurnEvent[]` | 禁止在此读 gameState |
| `src/core/gameState.ts` | 数据结构、存档读写、属性汇总 | 禁止在此写 UI 或战斗逻辑 |
| `src/data/xpTable.ts` | XP 升级阈值表（1-100 级） | 禁止写计算公式，只写预生成数值 |
| `src/data/dungeonTable.ts` | 副本/Boss 静态配置 | 禁止在游戏逻辑中动态生成 Boss |

## 2. 核心数值公式（S&F 原版复刻）

> **所有公式均在 `mathCore.ts` 中实现，UI 层只调用函数，不重复实现。**

### 2.1 生命值
`HP = 体质 (Constitution) × 等级 (Level) × 职业系数`

| 职业 | 代号 | 主属性 | HP 系数 | 护甲上限 |
|---|---|---|---|---|
| 猛将 (Warrior)  | CLASS_A | 武力 Strength | 5 | 50% |
| 游侠 (Scout)    | CLASS_B | 身法 Dexterity | 4 | 25% |
| 谋士 (Mage)     | CLASS_C | 智谋 Intelligence | 2 | 10% |
| 刺客 (Assassin) | CLASS_D | 身法 Dexterity | 4 | 25% |

### 2.2 伤害
`最终伤害 = 武器基础伤害 × (1 + 主属性 / 10)`  
刺客双持系数：`× 0.625`（每次攻击）

### 2.3 暴击率
`暴击率 = min(50%, 幸运 × 5 / (敌方等级 × 2))`

### 2.4 护甲减伤率
`减伤率 = min(职业上限, 护甲值 / 敌方等级)`

### 2.5 职业特殊机制
- **猛将 (CLASS_A)：** 装备盾牌时，每轮有 25% 概率格挡（闪避伤害为 0）
- **游侠 (CLASS_B)：** 每轮有 50% 概率闪避敌方攻击（非谋士攻击）
- **谋士 (CLASS_C)：** 攻击必中（无视闪避/格挡），并**无视护甲**
- **刺客 (CLASS_D)：** 每轮进行两次攻击（主手 + 副手武器），每次伤害 × 0.625

## 3. XP 与升级系统

### 3.1 设计基准：S&F 经典版 (2011)
- 经典版参考值：Lv1 需 400 XP，Lv10 需 6,435 XP，Lv100 约 300 万 XP
- **不使用现代加速版数据**（现代版在付费加速生态下膨胀，脱离付费后体验极差）

### 3.2 升级判断
- 读取 `src/data/xpTable.ts` 的 `XP_TABLE[currentLevel]` 得到升级所需经验
- 调用 `mathCore.ts` 中的 `checkLevelUp()` 函数，支持连续升级
- **等级上限：100 级**

### 3.3 经验来源（S&F 对齐原则）
- 主要来源：接取任务（`MissionHub`）
- 副本奖励：每个 Boss 给约 **8-15%** 的当前等级升级所需 XP（章节最终 Boss 给 15%）

## 4. 副本系统规范

### 4.1 设计基准
- 每个章节固定 **10 个 Boss**，按顺序挑战
- 章节依**玩家等级**解锁（存储在 `chapter.unlockLevel`）
- 进度存储在 `gameState.dungeonProgress: Record<string, number>` 中

### 4.2 挑战次数限制（S&F 经典版机制）
- **每日 1 次免费尝试**，午夜 00:00 刷新
- 次数记录在 `gameState.dungeonDailyAttempt: { date: string; used: number }`

### 4.3 Boss 数值字段说明
- `attributes.dexterity` = 身法/Agility（在战斗引擎中映射为 `agility`）
- `weaponDamage` 为基础面板，实际出伤范围为 `[面板×0.7, 面板×1.3]`

## 5. UI / UX 规范

### 5.1 视觉风格
- 深色调、高对比度硬核放置风格
- 主题色：深红 `#b02a2a`，背景 `#121212`
- 字体：Inter / 系统字体回退

### 5.2 战斗表现层（`Dungeon.tsx`）
参照 `docs/05_UI_Combat.md` 完整规范，核心要点：
- 战斗结果在逻辑层（`battleCore`）**瞬间计算**完毕，返回 `TurnEvent[]`
- 前端以 700ms/帧 的节奏回放事件序列
- 必须实现：平滑血条、浮动伤害跳字（FCT）、武器闪击特效、跳过按钮、胜利/失败结算 Modal

### 5.3 透明化原则（数值显示）
- 玩家面板中**所有二级属性（HP/伤害/护甲/暴击率）必须实时调用 `mathCore` 计算**
- 鼠标悬浮 Tooltip 展示底层计算公式字符串，绝对禁止 UI 层写死计算

## 6. 文件编码规范

> ⚠️ **PowerShell 写文件时中文字符会乱码，务必避免。**

- 所有 TypeScript/TSX 源文件用 **`write_to_file` 工具写入**（非 PowerShell Set-Content）
- 数据文件（`src/data/*.ts`）如须通过 PowerShell 写入，中文内容必须单独处理
- 发现乱码时，用 `view_file` 工具验证，用 `multi_replace_file_content` 精准修复

---

## 7. 存档迁移策略

每次修改 `GameState` 接口结构时：
1. 在 `loadGameState()` 中对新字段提供默认值回退（`parsedState.newField ?? defaultValue`）
2. 如有破坏性变更（字段删除/类型变更），设置 `needsWipe` 标志清空相关字段
3. 开发阶段可通过导航栏「销毁存档，重新来过」按钮强制清档测试
